# Sample 05 — Flutter App

Demonstrates two developer workflows for consuming Bella Baxter secrets in Flutter:
using compile-time dart-defines (**dev mode**) and the production pattern with
`BellaClient` runtime fetch + encrypted local cache (**runtime mode**).

---

## Why `bella exec -- flutter run` doesn't work

`String.fromEnvironment('KEY')` is a **compile-time constant** in Dart/Flutter.
It is only populated via `--dart-define=KEY=VAL` flags at build/run time — **not**
from shell environment variables.

`bella exec` injects secrets into the shell process environment, but Flutter
ignores those for `String.fromEnvironment`. The only supported mechanism is
`--dart-define` or `--dart-define-from-file`.

---

## Flow A — Dev mode (no API key required)

Best for local development: zero API calls, works offline, no pay-per-use.

Secrets are baked in at compile time as dart-define constants.

### Setup

```bash
# 1. Fetch your app secrets as dart-define-compatible JSON:
bella get secrets --format json -o .secrets.json

# 2. Run the app — secrets come from compile-time dart-defines:
flutter run \
  --dart-define=BELLA_DEV_MODE=true \
  --dart-define-from-file=.secrets.json
```

Or use the **"Dev mode (no API key)"** launch config in VS Code.

### How it works

```
main()  ──► _devMode = true
         ──► AppSecrets.devMode()  (no network call)
              └─► getters read from static const String.fromEnvironment(...)
                  baked in at compile time from .secrets.json
         ──► runApp(MyApp(client: null, ...))
```

The app displays your secrets. A blue `🛠` icon in the toolbar indicates dev mode.

---

## Flow B — Runtime mode (BellaClient, production simulation)

Best for testing the production code path: `BellaClient` fetches secrets at app
startup and caches them encrypted in `FlutterSecureStorage`.

### Setup

```bash
# 1. Copy the example file:
cp .env.example .env

# 2. Edit .env — fill in your Bella API key from the Bella dashboard:
#    { "BELLA_BAXTER_URL": "...", "BELLA_BAXTER_API_KEY": "bax-..." }

# 3. Run the app:
flutter run --dart-define-from-file=.env
```

Or use the **"Runtime mode (BellaClient)"** launch config in VS Code.

### How it works

```
main()  ──► BellaClient.pullSecretsAs(AppSecrets.fromMap)
              ├─► reads from FlutterSecureStorage cache (instant, offline-safe)
              └─► fetches fresh secrets from Bella API in background
         ──► runApp(MyApp(client: client, ...))
         ──► SecretsPage polls every 5 min via watchSecretsAs()
              └─► on each fetch: updates UI + writes back to encrypted cache
```

A green ☁ icon indicates fresh secrets; orange ☁✗ indicates offline (cached).

---

## Offline behaviour (runtime mode only)

On first launch, the app fetches secrets from the API and caches them in the
OS secure enclave:

| Platform | Storage |
|----------|---------|
| Android  | Android Keystore (AES-256) |
| iOS/macOS | Keychain |
| Linux    | libsecret / GNOME Keyring |
| Windows  | Windows Credential Manager |

Subsequent launches are instant and work without connectivity.

## Files

| File | Description |
|------|-------------|
| `.env` | Your Bella credentials (gitignored — never commit) |
| `.env.example` | Template for `.env` (safe to commit) |
| `.secrets.json` | Your app secrets for dev mode (gitignored — never commit) |
| `.secrets.json.example` | Template for `.secrets.json` (safe to commit) |
| `lib/app_secrets.dart` | Typed secrets class (auto-generated by `bella secrets generate dart --types`) |
| `lib/flutter_secure_secret_cache.dart` | `FlutterSecureStorage`-backed `SecretCache` |

---

## Regenerate typed secrets class

After changing your secrets in Bella, regenerate the typed accessor class:

```bash
bella secrets generate dart --types \
  -p my-project -e production \
  -o lib/app_secrets.dart
```
